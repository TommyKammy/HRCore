# Product

## Register

product

## Users

HR operators, approvers, HR Ops/support reviewers, and bounded admins use HRCore to rehearse employee lifecycle work against repository-owned synthetic or explicitly approved non-production records. They need direct work queues, clear state, and explicit evidence boundaries rather than broad employee search or production-console affordances.

## Product Purpose

HRCore provides an open-source HR core foundation for identity-driven onboarding, transfer, termination, approval, audit, and HR-to-IdP automation workflows. In the current phase, success means a bounded browser shell and API contract connection that help non-engineers inspect planned practical-use areas while keeping real employee data, live providers, production authorization, production audit immutability, broad export, retention/deletion runtime, legal/privacy approval, two-key approval, and go-live readiness blocked.

## Brand Personality

Careful, explicit, and operational. The product voice should feel trustworthy for regulated HR work without implying production readiness before the required authority exists.

## Anti-references

Avoid SaaS marketing dashboards, production admin consoles, broad employee directory patterns, unrestricted search/export surfaces, decorative analytics, and any wording that presents bounded synthetic evidence as real HR practical-use or production-like readiness.

## Design Principles

- Start from work queues and anchored records, not global search.
- Show bounded/non-production context wherever authority could be mistaken for production access.
- Fail closed when persona, tenant, scope, subject, or evidence signals are missing.
- Keep derived summaries subordinate to authoritative lifecycle and evidence records.
- Make empty, loading, and error states explain the blocked boundary without offering bypasses.

## Accessibility & Inclusion

Target WCAG 2.1 AA for the browser shell. Use semantic navigation and landmarks, visible focus states, reduced-motion-safe transitions, readable contrast, and route smoke coverage that can be extended with deeper accessibility checks as workflows mature.
